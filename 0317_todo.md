# 0317 迭代计划

> 日期：2026-03-17
> 项目：虾虾工厂 / `factoryxiaxia`

## 一、当前项目判断

当前仓库已经具备以下基础能力：

- 前台：官网、登录注册、员工列表、员工详情、下单、支付、订单、消息
- 后台：数据看板、用户管理、员工管理、订单管理
- 后端：Flask API + MySQL + JWT + Docker Compose
- 测试：已有一套 API 测试，但本地运行环境还未补齐依赖

当前最主要的问题不是“没有功能”，而是：

- 有些页面已经接真实 API，有些页面仍然是演示稿式静态内容
- 后端业务约束不够严，存在状态流转和权限边界风险
- 文档、部署、配置存在不一致和默认值过于宽松的问题
- 后端仍是单文件结构，后续继续堆功能会越来越难维护

## 二、本轮迭代目标

本轮迭代的核心目标分为四层：

1. 先把能出问题的地方补齐，降低线上风险
2. 再把已经写出来但没真正接通的前端页面产品化
3. 然后整理工程结构，让后续开发不再堆在一个文件里
4. 最后补齐测试、部署、文档，形成稳定交付闭环

## 三、迭代拆分

### 迭代 1：稳定性修补

**目标**

先处理高风险问题，保证“能跑”和“不会明显跑歪”。

**任务**

- [x] 登录接口补充禁用用户校验，禁止 `is_active = false` 的账号继续登录
- [x] 管理后台修改订单状态时，改为复用统一状态机校验，而不是允许任意跳转
- [x] 补充订单状态变更对应时间戳处理逻辑（新增 `activated_at` 字段和时间戳设置）
- [x] 梳理退款、取消、完成、激活流程中的边界条件
- [x] 校正文档和部署脚本中的前端访问端口说明（统一为 `localhost:10088`）
- [x] 明确 `.env.example` 的使用方式，补充生产环境配置说明
- [x] 处理默认管理员账号和默认密钥的风险提示（docker-compose.yml 添加注释）

**交付物**

- 后端关键业务修复
- 更新后的部署文档
- 更清晰的环境配置说明

**验收标准**

- 禁用账号无法登录
- 管理员无法绕过状态机直接写入非法订单状态
- `README.md`、`DEPLOY.md`、`deploy.sh`、`docker-compose.yml` 中访问地址保持一致

### 迭代 2：前端产品化补全

**目标**

把当前“半成品页面”补成真实可用页面，优先提升用户端主流程完整度。

**任务**

- [x] 个人中心改为真实数据驱动，移除明显演示数据
- [x] 个人中心接入订单摘要、未读消息数、最近订单、推荐员工
- [x] 员工详情页接入评价列表（含分页、星级展示）
- [x] 员工详情页接入收藏状态、收藏/取消收藏操作（心形图标切换）
- [x] 员工详情页接入相关推荐（4个推荐员工卡片）
- [x] 支付成功后形成更顺畅的回跳路径（下单→支付页→5秒倒计时→订单详情）
- [x] 订单详情页补充评价入口（星级选择+文本输入，已评价则展示）
- [x] 检查各页面登录态失效后的统一跳转与提示（pay/order-create/profile/worker-detail/order-detail 统一 401 处理）

**交付物**

- 完整可用的 `profile.html`
- 补全后的 `worker-detail.html`
- 用户主链路：浏览 -> 下单 -> 支付 -> 查看订单 -> 评价

**验收标准**

- 用户在个人中心看到的核心数据全部来自 API
- 员工详情页可以完成“看评价 + 收藏 + 下单”的闭环
- 登录态过期时不会停留在错误页面

### 迭代 3：后台与业务能力整理

**目标**

把后台从“能操作”提升到“更适合运营使用”，并补齐业务管理缺口。

**任务**

- [x] 后台订单管理补充更明确的状态流转提示（仅展示合法下一状态，终态显示"终态"标签）
- [x] 后台订单管理补充筛选项和操作反馈（toast 通知替代 alert，确认弹窗）
- [x] 后台用户管理补充禁用原因或操作备注（禁用前确认弹窗，toast 反馈）
- [x] 后台员工管理补充字段校验和异常提示（名称/费率/等级/分类必填校验，toast 反馈）
- [x] 梳理消息触发点，确保核心动作均能生成站内消息（补充 activate、review、admin 状态变更的 send_message）
- [x] 统一前后台错误提示风格（全站 alert 替换为 toast 通知）

**交付物**

- 更稳的后台管理体验
- 更完整的站内通知链路

**验收标准**

- 后台常见操作有明确成功/失败反馈
- 后台无法产生明显脏数据
- 核心业务动作能触达站内消息

### 迭代 4：后端结构重构

**目标**

把单文件后端拆分为可持续维护的结构，为后续继续开发做准备。

**任务**

- [x] 按领域拆分 `app.py`（拆为 extensions/models/routes/services/utils 共 11 个模块）
- [x] 抽离模型层、认证层、订单服务层、管理后台层
- [x] 抽离通用工具函数，如 token、消息、订单号生成、状态机校验
- [x] 统一错误返回格式（各蓝图独立处理，app.py 仅处理全局 500/429）
- [x] 为后续引入迁移工具预留结构（models.py 独立，extensions.py 延迟初始化）
- [x] 清理重复查询和重复序列化逻辑

**建议拆分方向**

- `backend/app.py` 仅保留应用初始化
- `backend/models.py`
- `backend/routes/auth.py`
- `backend/routes/catalog.py`
- `backend/routes/orders.py`
- `backend/routes/admin.py`
- `backend/services/orders.py`
- `backend/services/messages.py`
- `backend/utils/auth.py`

**交付物**

- 拆分后的后端目录结构
- 更清晰的模块边界

**验收标准**

- 主流程功能不回归
- 单文件不再承载全部模型和全部路由
- 后续新增接口可以按模块继续扩展

## 四、推荐执行顺序

建议不要并行乱做，按照下面顺序推进：

1. 迭代 1：先修风险和文档不一致
2. 迭代 2：补前端真实数据与主流程体验
3. 迭代 3：整理后台操作体验和业务闭环
4. 迭代 4：开始后端模块拆分
5. 迭代 5：补测试和交付保障

## 五、建议本周优先开工项

- [ ] 修复禁用用户仍可登录的问题
- [ ] 修复后台订单状态任意修改的问题
- [ ] 统一部署端口和文档说明

## 六、完成定义

当下面几条都满足时，可以认为项目进入“可持续迭代”状态：

- 主链路功能可真实使用，不再依赖演示数据撑页面
- 关键权限和状态流转已加上约束
- 部署方式、访问地址、环境变量说明全部一致
- 测试可以在新环境复现运行
- 后端结构不再依赖一个超大单文件
